# CLAUDE.md — конституция базы знаний SGCapital Wiki

Это «схема» в смысле паттерна Карпаты LLM Wiki
(https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).
Файл описывает, как устроена эта база знаний, какие у неё правила, и
какие рабочие процедуры должен выполнять LLM-агент, когда он с ней
работает. Человек и агент дорабатывают этот файл совместно по мере роста
базы.

## 1. О чём база

Корпус — многолетний экспорт Telegram-канала **«Капитал» (SGCapital,
t.me/sgcapital)**. Один автор, **1996 сообщений** с **2017-04-04** по
**2026-04-14**, почти 9 лет наблюдений за рынками и размышлений о
личных финансах и инвестировании.

Цель базы — превратить этот чат в **компактную, структурированную,
самоподдерживающуюся память**, из которой и автор-читатель, и
LLM-агент могут быстро доставать обобщённый опыт и конкретные цитаты с
датами.

## 2. Три слоя (Карпаты)

### 2.1. Raw sources — **неизменяемые**

В эти файлы агент **никогда не пишет**, только читает.

- `messages.html`, `messages2.html`, `messages3.html` — оригинальный
  Telegram-экспорт.
- `files/*.pdf`, `photo_*.jpg`, `photo_*_thumb.jpg` — вложения из
  экспорта.
- `css/`, `js/`, `images/` — оформление экспорта. Не трогать.
- `tools/chat.jsonl` — производная от HTML, но тоже **только
  пересобирается скриптом** (`tools/parse_chat.py`); руками не
  редактируется. Считается raw-слоем для быстрых grep-запросов.

**Правило:** если нужно изменить содержимое raw-слоя — меняется
исходный HTML (чего мы не делаем) или скрипт-парсер, но не JSONL
напрямую.

### 2.2. Wiki — **LLM-owned**

Директория `wiki/`. Весь markdown там пишется и поддерживается
LLM-агентом в диалоге с пользователем. Структура:

```
wiki/
├── index.md           # каталог всех страниц с однострочными описаниями
├── log.md             # хронологический append-only лог
├── overview.md        # обзор канала и синтез верхнего уровня
├── concepts/          # принципы и концепции (как думать)
├── entities/          # сущности и инструменты (что покупать/наблюдать)
├── theses/            # «сквозные тезисы» автора и их эволюция
├── timeline/          # ключевые периоды и события
└── queries/           # сохранённые ответы на вопросы пользователя
```

### 2.3. Schema — этот файл

`CLAUDE.md` в корне. Живой документ, правится совместно.

## 3. Что в эту базу НЕ входит (scope filter)

Пользователю **не нужны** следующие темы, и в `wiki/` по ним страниц
**не создаётся**, а при ingest они пропускаются:

- Рубль, курсы USD/RUB, EUR/RUB, девальвации рубля.
- Российский рынок акций (Мосбиржа, IMOEX, РТС, Сбербанк, Газпром,
  Лукойл и прочие российские эмитенты).
- ОФЗ, рублёвые корпоративные облигации, ключевая ставка ЦБ РФ.
- ИИС и российские налоговые льготы.
- Локальная российская недвижимость.

Если сообщение затрагивает и глобальный, и российский аспект —
цитируется **только глобальная часть**, а RU-специфичные куски
игнорируются.

Критерии автоматического отсеивания сообщений из ingest (реализованы в
`tools/build_wiki.py` как `RU_HARD_TERMS`): «рубл», «офз», «мосбирж»,
«ммвб», «сбербанк», «газпром», «лукойл», «роснефт», «норникел», «втб»,
«иис», «цб рф», «банка росс», «ключевая ставка».

## 4. Формат страницы вики

Каждая страница `concepts/*.md`, `entities/*.md`, `theses/*.md`,
`timeline/*.md` должна иметь:

1. **YAML frontmatter** с полями:
   ```yaml
   ---
   title: "Название страницы"
   type: concept | entity | thesis | timeline
   tags: [tag1, tag2]
   first_seen: YYYY-MM-DD   # самая ранняя цитата
   last_seen:  YYYY-MM-DD   # самая поздняя цитата
   source_count: <N>        # сколько сообщений корпуса цитируется
   updated: YYYY-MM-DD
   ---
   ```
2. **Краткое резюме** (2–6 предложений) — что автор думает по теме
   одним абзацем. Пишется LLM-агентом.
3. **Ключевые идеи** — bullet-список из 3–8 пунктов с опорой на
   конкретные сообщения.
4. **Эволюция взглядов** — если позиция автора менялась во времени,
   явно показываем «раньше → позже» с датами.
5. **Цитаты** — хронологическая подборка сообщений с обязательными
   метаданными: дата ISO + ссылка `messages.html#messageN` (или
   `messages2.html#...`, `messages3.html#...`).
6. **Связи** — секция «См. также» с внутренними ссылками на другие
   страницы вики.

## 5. Правила работы LLM-агента

### 5.1. Источники и цитирование

- **Любое утверждение о взглядах автора** сопровождается ссылкой на
  конкретное сообщение корпуса в формате
  `([YYYY-MM-DD](messages.html#messageN))`.
- Разделять **прямую цитату** (блок `> ...`) и **синтез LLM** (обычный
  текст с пометкой «[синтез]»). Синтез должен быть узнаваем, чтобы его
  можно было ревьюить отдельно от цитат.
- Никогда не выдумывать цитаты и даты. Если источник не найден — так и
  писать: «источник не найден».

### 5.2. Raw-слой неприкосновенен

- Не редактировать `messages*.html`, `files/*.pdf`, `photo_*.jpg`,
  `chat.jsonl` напрямую.
- Изменения в raw-слой возможны только через перезапуск
  `tools/parse_chat.py` поверх обновлённых HTML.

### 5.3. Язык

- Единственный язык вики — **русский**. Английские термины (ETF,
  rebalancing, S&P 500) оставляем как есть, где это принято.

### 5.4. Эволюция взглядов, не перезапись

- Если новая запись меняет позицию автора, **старая секция не
  удаляется**. Вместо этого в «Эволюция взглядов» добавляется новый
  пункт с датой. База знаний должна помнить, что думали год назад.

### 5.5. Лог (append-only)

- После каждой значимой операции (ingest / query / lint) агент
  дописывает запись в `wiki/log.md`. Формат префикса строгий:
  ```
  ## [YYYY-MM-DD] ingest | <что>
  ## [YYYY-MM-DD] query  | <что спросили>
  ## [YYYY-MM-DD] lint   | <что проверяли>
  ```
  Такой префикс парсится тривиальным grep (`grep "^## \[" wiki/log.md`).

### 5.6. Индекс

- `wiki/index.md` — единственный маршрутный файл. При добавлении
  страницы агент **обязан** обновить `index.md` и добавить
  однострочное описание.
- При query сначала читается `wiki/index.md`, затем выбранные
  страницы, только потом — при необходимости — raw-слой через grep по
  `tools/chat.jsonl`.

### 5.7. Cross-refs

- Каждая страница должна иметь минимум **2 исходящие ссылки** на
  другие страницы вики (если это возможно для темы).
- Висячих (orphan) страниц без входящих ссылок быть не должно —
  ловится во время lint.

## 6. Рабочие процедуры

### 6.1. Ingest (появление новых сообщений в канале)

1. Обновить исходные `messages*.html` экспортом из Telegram.
2. `python3 tools/parse_chat.py` — перестроить `tools/chat.jsonl`.
3. `python3 tools/build_index.py` — обновить `INDEX.md`
   (тематический указатель всех сообщений, вспомогательный view).
4. `python3 tools/build_wiki.py --since YYYY-MM-DD` — дописать новые
   цитаты в существующие страницы `wiki/`; для каждой тронутой
   страницы пересчитать `last_seen` и `source_count`.
5. Агент проходится по тронутым страницам, обновляет резюме и
   «Эволюция взглядов», если позиция изменилась.
6. Запись в `wiki/log.md`:
   `## [YYYY-MM-DD] ingest | <количество новых сообщений>, <кратко>`.

### 6.2. Query (пользователь задаёт вопрос)

1. Агент читает `wiki/index.md`.
2. Выбирает 1–5 релевантных страниц, читает их целиком.
3. При недостатке данных — grep по `tools/chat.jsonl`.
4. Ответ оформляется с цитатами и ссылками
   `messages.html#messageN`.
5. **Хорошие ответы сохраняются** в `wiki/queries/<YYYY-MM-DD>-<slug>.md`,
   чтобы исследование компаундировалось. Не всё подряд — только то, что
   представляет ценность для будущих вопросов.
6. Запись в `wiki/log.md`:
   `## [YYYY-MM-DD] query | <вопрос>`.

### 6.3. Lint (здоровье базы)

Периодически (например, раз в месяц или перед каждым крупным заходом)
агент проверяет:

- **Orphan pages** — страницы без входящих ссылок из других страниц
  или из `index.md`.
- **Контрадикции** — места, где страницы противоречат друг другу или
  где thesis устарел относительно более свежих сообщений.
- **Concept gaps** — термины, часто встречающиеся в корпусе или в
  других страницах, но не имеющие своей страницы.
- **Отсутствующие cross-refs** — упоминания темы X на странице Y без
  ссылки на `wiki/.../x.md`.
- **Stale `last_seen`** — страницы, чей `last_seen` отстаёт от даты
  последнего сообщения в корпусе, хотя тема продолжает упоминаться.

Результат оформляется одной записью в `wiki/log.md`:
`## [YYYY-MM-DD] lint | <N находок, ссылки>`, и конкретные правки
вносятся в тронутые страницы.

## 7. Тонкие правила

- **Картинки.** LLM не читает markdown со встроенными изображениями за
  один проход. Если на странице есть ссылки на `photo_*.jpg` из
  корпуса, агент сначала читает текст, затем отдельно открывает
  нужные картинки.
- **Длина цитат.** Полный текст сообщения включается, если он ≤ 600
  символов. Иначе цитируется осмысленный фрагмент с пометкой «[…]»
  на месте сокращения.
- **Пагинация.** Если раздел `Цитаты` на странице разрастается выше
  ~50 записей, часть старых группируется под свёрткой «Архив цитат»
  или выносится в отдельный файл `<slug>-archive.md`.

## 8. Инструменты в `tools/`

- `tools/parse_chat.py` — HTML → `tools/chat.jsonl`.
- `tools/build_index.py` — `chat.jsonl` → `INDEX.md` (тематический
  индекс сообщений; вспомогательный view, не часть вики).
- `tools/build_wiki.py` — `chat.jsonl` → скелет страниц `wiki/` с
  цитатами по спецификациям страниц. Сохраняет ручные секции
  (frontmatter, резюме, «Эволюция взглядов», «Ключевые идеи», «См.
  также») между маркерами; регенерирует только блок цитат.

Маркер сохранения ручного контента в страницах:
```
<!-- autogen:quotes:start -->
...автогенерируемый блок цитат...
<!-- autogen:quotes:end -->
```
Весь текст вне этих маркеров — ручной и скриптом не трогается.
